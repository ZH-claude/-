# 2026-06-19 线路计费纠偏记录

## 正确业务顺序

这次重新锁定商家端方向，不能再把上游接入、模型发布、模型映射混在一个入口里。

1. 先发布客户看到的模型，例如 `gpt5.5`。
2. 再分别接入上游，例如 DeepSeek 上游、中转站上游。
3. 最后在模型映射里，把 `gpt5.5` 绑定到一条或多条上游线路。
4. 用户请求 `gpt5.5` 时，系统按线路顺序尝试；哪条线路最终成功，就按哪条线路的扣费规则扣 token。

## 两类上游的计费口径

DeepSeek 上游：
- DeepSeek 普通模型按 1 倍 token 计费。
- 其他公开模型可以设置倍率，例如 `gpt5.5` 设置为 5 倍。
- 真实消耗 token 先按上游返回的输入和输出 token 统计，再乘以模型倍率。

中转站上游：
- 商家输入上游的输入价格、输出价格、汇率和加价百分比。
- 系统自动加 10% 差价，再换算成统一 token 扣费倍率。
- 该线路成功时，按这条线路自己的价格规则扣 token。

## 保留风险

- 如果真实上游没有返回用量数字，系统不能乱扣费，会标记为计量未知。这是保护用户余额的正确行为。
- 旧数据库里仍有历史字段名带金额含义，当前业务已经按 token 口径扣费；后续商用整理时应统一字段命名，避免维护人员误解。
