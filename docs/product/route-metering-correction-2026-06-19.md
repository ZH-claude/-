# 2026-06-19 线路计费纠偏记录

## 正确业务顺序

这次重新锁定商家端方向，不能再把上游接入、模型发布、模型映射混在一个入口里。

1. 先发布客户看到的模型，例如 `gpt5.5`。
2. 再分别接入上游，例如 DeepSeek 上游、中转站上游。
3. 最后在模型映射里，把每个客户模型绑定到唯一一条上游线路。
4. 用户请求某个客户模型时，后台只走该模型绑定的上游线路，并按这条线路的规则扣费；用户端不显示真实上游信息。

## 两类上游的计费口径

DeepSeek 上游（统一口径）：
- DeepSeek 普通模型按 1 倍价格基准计费，token 数量仍按上游真实返回显示。
- 其他公开模型可设置商家价格倍率，例如 `gpt5.5` 设置为 5 倍。
- 真实消耗按上游返回的输入/输出 token 原样统计；token 不乘倍率，扣费金额按该模型价格倍率计算。

中转站上游：
- 商家输入上游的输入价格、输出价格、汇率和加价百分比。
- 系统以商家输入/输出价格为基准先自动加 10% 差价，再按真实输入/输出 token 计算扣费金额。
- 该线路成功时，token 按真实消耗记录，金额按该线路的价格规则计算。

## 保留风险

- 如果真实上游没有返回用量数字，系统不能乱扣费，会标记为计量未知。这是保护用户余额的正确行为。
- 旧数据库里仍有历史字段名带金额含义，当前业务已经按 token 口径扣费；后续商用整理时应统一字段命名，避免维护人员误解。
