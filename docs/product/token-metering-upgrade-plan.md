# 统一 token 计费升级规划

更新时间：2026-06-18

## 目标

把当前“用户端按美元余额显示、商家端按人民币显示”的旧计费方式，升级为统一的“token 额度”方式。

升级后，用户不再看美元余额；用户看到的是自己还剩多少 token。商家仍然可以按人民币生成充值码，因为商家收钱和核算成本需要人民币口径。

## 当前锁定方向

这次方向纠偏后，商家端必须按三步走，后续开发和测试都按这个顺序验收：

1. 先在 `模型发布` 准备客户看到的模型，例如 `gpt5.5`。这里不接上游、不填真实上游模型、不算线路扣费。
2. 再到 `DeepSeek 上游` 或 `中转站上游` 接入供应商。这里只保存上游名称、地址、密钥、状态和健康检查。
3. 最后到 `模型映射` 把客户模型绑定到唯一一条上游线路。这里才填写真实上游模型、超时时间、提示词、是否流式输出和线路计费规则。

如果某个模型报错，优先检查第三步的线路绑定是否正确，不能回到上游页面把供应商当成模型重新发布。

## 核心规则

| 项目 | 新规则 |
| --- | --- |
| 用户余额 | 统一显示为 token，不再显示美元余额 |
| 充值码 | 商家按人民币生成，用户兑换后得到 token |
| 默认兑换率 | 约 8 元 = 100 万 token |
| 基准模型 | 普通 DeepSeek 按 1 倍消耗 |
| 其他模型 | token 按真实用量显示，金额按模型价格倍率计算 |
| 失败调用 | 不扣用户 token |
| 成功调用 | 必须产生真实调用记录、真实用量记录、真实扣费流水 |
| 上游成本 | 商家端单独记录，用来算利润，不直接给用户看 |

示例：

| 模型 | 计费理解 |
| --- | --- |
| DeepSeek 普通模型 | 上游返回 1,000 token，用户端显示 1,000 token，金额按 1 倍价格扣 |
| GPT-5 | 上游返回 1,000 token，用户端显示 1,000 token，金额按 5 倍价格扣 |
| Claude Opus 4.8 | 上游返回 1,000 token，用户端显示 1,000 token，金额按 8 倍价格扣 |

## 两种上游要分开配置

不要把两种上游全部挤在一个页面里。商家端要拆成两个清楚入口。

操作顺序必须拆成三段：

1. `模型发布`：只准备客户看到的模型名，例如 `gpt5.5`。
2. `DeepSeek 上游` / `中转站上游`：只保存上游地址、密钥、状态和健康检查。
3. `模型映射`：才把客户模型绑定到真实上游线路，并设置真实上游模型名、超时、附加提示词和计费规则。一个客户模型只启用一条上游线路。

### 入口一：DeepSeek 官方上游

用途：适合把 DeepSeek 作为基础 1 倍模型。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 名称 | 商家自己识别这条上游，例如 deepseek-v4-pro |
| 上游地址 | DeepSeek 官方接口地址 |
| 上游密钥 | 商家自己的 DeepSeek 密钥 |
| 状态 | 启用或停用 |
| 健康检查 | 检查地址和密钥是否可用 |

验收标准：

- DeepSeek 页面不发布客户模型。
- DeepSeek 页面不绑定真实上游模型名。
- DeepSeek 页面只做上游接入和检查。

### 入口二：其他中转站上游

用途：适合接入你从别的中转站拿到的价格表。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 名称 | 商家自己识别这条上游，例如中转站 1 |
| 上游地址 | 别的中转站接口地址 |
| 上游密钥 | 别的中转站给你的密钥 |
| 状态 | 启用或停用 |
| 健康检查 | 检查地址和密钥是否可用 |

中转站页面不填写客户模型名，也不填写真实模型名。输入价格、输出价格、货币、汇率、加价比例放到 `模型映射` 里，因为同一个中转站上游可能服务多个客户模型。

`模型映射` 里的倍率计算规则：

```text
1 倍 token 的成本基准 = 8 元 / 100 万 token

如果上游是人民币价格：
倍率 = 上游每 100 万 token 价格 * 1.10 / 8

如果上游是美元价格：
倍率 = 上游每 100 万 token 价格 * 汇率 * 1.10 / 8
```

图片里的示例价格：

| 项目 | 上游价格 |
| --- | --- |
| 输入 | 5 美元 / 100 万 token |
| 输出 | 30 美元 / 100 万 token |

如果商家填写汇率 7.2，那么系统应自动算出：

| 项目 | 计算 |
| --- | --- |
| 输入倍率 | 5 * 7.2 * 1.10 / 8 = 4.95 倍 |
| 输出倍率 | 30 * 7.2 * 1.10 / 8 = 29.70 倍 |

这里不写死汇率。商家输入多少汇率，系统就按多少汇率算。

## 用户端显示规则

| 页面 | 新显示 |
| --- | --- |
| 个人中心 | 剩余 token、累计消耗 token、调用次数 |
| 余额充值 | 输入充值码后显示到账多少 token |
| 令牌管理 | 显示令牌已消耗 token、剩余额度 |
| 调用日志 | 显示本次上游真实输入 token、输出 token、总 token 和实际扣费金额 |
| 费用说明 | 解释 token 原始用量和美元扣费金额，不把价格倍率写成 token 放大 |

用户端要让普通用户看得懂：

- 充值码兑换的是 token，不是美元。
- DeepSeek 是 1 倍价格。
- 其他模型是几倍价格，就按几倍金额扣费。
- 用户能看到“这次用了多少真实 token、实际扣了多少钱”。

## 商家端显示规则

| 页面 | 新显示 |
| --- | --- |
| 商家首页 | 客户剩余 token、今日消耗 token、今日预估人民币收入 |
| 充值码 | 按人民币生成充值码，同时显示可兑换 token |
| DeepSeek 上游 | 独立页面，适合 1 倍基准模型 |
| 中转站上游 | 独立页面，只保存供应商地址、密钥、状态和健康检查 |
| 模型发布 | 只管公开模型名、显示名、启停 |
| 模型映射 | 只管公开模型绑定到哪个上游、哪条线路，并在线路里设置扣费 |
| 请求日志 | 显示真实上游、真实用量、扣费金额、上游成本、平台毛利 |

商家端必须保留人民币口径，因为充值码、成本、利润都要按人民币核算。

- 不能再让用户端继续围绕美元余额理解消费。

## 2026-06-19 流程纠偏

正确业务顺序已经统一为三步：

1. 先发布客户模型：例如先在 `模型发布` 页面发布客户看到的 `gpt5.5`，这里只决定用户端能看到什么模型名和状态。
2. 再接入上游供应商：`DeepSeek 上游` 只保存 DeepSeek 的上游地址、密钥、状态和健康检查；`中转站上游` 只保存其他中转站的上游地址、密钥、状态和健康检查。
3. 最后绑定上游线路：在 `模型映射` 页面把每个客户模型绑定到唯一一条上游线路，例如后台可以让 `gpt5.5 低消耗` 绑定 DeepSeek，让 `gpt5.5 中消耗` 绑定中转站 1。这里填写真实上游模型名、超时时间、附加提示词、线路计费方式和是否支持流式输出。

边界规则：

- 上游页面不能直接发布客户模型，也不能替代模型发布页面。
- 一个客户模型只能启用一条上游线路。需要不同成本、不同质量或不同供应商时，必须发布成不同的客户模型，避免同一个模型背后偷偷切换路线。
- `gpt5.5` 这类客户模型是对用户展示的名称；真实请求走哪个上游、真实上游模型叫什么，只在商家端和后台保存，普通用户看不到。
- DeepSeek 上游和其他中转站上游可以有不同计费方式，但 token 只展示真实消耗，扣费金额按该模型的价格规则计算。
- 后续继续开发时，任何页面、接口、自检脚本都必须按以上顺序验收，不能再把“上游接入”和“客户模型发布”混在同一个入口里。

这次纠偏后的目标不是在上游页面直接发布模型，而是把三个东西彻底分开：

| 步骤 | 商家看到的动作 | 保存的真实含义 | 验收标准 |
| --- | --- | --- | --- |
| 1 | 发布客户模型，例如 `gpt5.5` | 只定义用户端可见的模型商品 | 用户端只在后台绑定可用线路后看到该商品名，不显示线路 |
| 2 | 接入 DeepSeek 上游或中转站上游 | 只保存供应商地址、密钥、状态、健康检查 | 上游列表能查看、修改、检查，不发布客户模型 |
| 3 | 模型映射选择上游线路并配置扣费 | 把客户模型绑定到唯一上游、真实上游模型、超时、提示词和扣费规则 | 一个客户模型只启用一条线路；如果要低消耗和中消耗，就发布两个客户模型，后台分别绑定 DeepSeek 或中转站 |

## 2026-06-19 三次纠偏：用户端按商品名展示扣费，不展示线路

本次确认的真实问题：

- 运行时请求会按客户模型绑定的唯一线路选择上游。
- 旧逻辑里，对外可用模型列表可能只看 `模型发布` 时的默认价格，没有优先看唯一启用线路的扣费规则。
- 如果业务上需要 DeepSeek 和中转站两种成本档位，必须拆成两个客户模型分别展示，否则商家和用户都会误判成本。

已完成：

- [x] 对外可用模型列表优先读取唯一启用线路的扣费规则；没有线路规则时才退回客户模型默认值，但不返回上游供应商名称。
- [x] 已绑定线路的上游不允许修改上游类型，避免 DeepSeek 线路被改成中转站线路后旧计费规则失效。
- [x] 自检脚本已覆盖两个独立客户模型分别绑定 DeepSeek 和中转站上游的情况，确认用户端只看到两个商品名，不暴露后台线路，也不会让同一个模型同时走两条线路。

当前保留的操作要求：

- 老线路如果以前没有保存线路级扣费规则，需要商家进入 `模型映射`，点该线路的 `修改`，重新保存真实上游模型名、倍率或中转站价格。
- 如果同一个客户模型曾经配置过多条线路，只保留当前启用的一条；另一条应该新建成独立客户模型，例如 `gpt5.5 低消耗` 或 `gpt5.5 中消耗`。
- 新发布的客户模型不应在 `模型发布` 页面配置价格；价格必须在线路里配置。
- 后续对账页面必须继续按线路、请求、令牌、用户余额四处一致来验收。

## 2026-06-19 四次纠偏：上游页只管供应商

本次纠偏锁定的新边界：

1. `模型发布` 是第一步，只发布客户能看到的模型，例如 `gpt5.5`。
2. `DeepSeek 上游` 和 `中转站上游` 是第二步，只接入供应商地址、密钥、状态和健康检查。
3. `模型映射` 是第三步，才把 `gpt5.5` 绑定到 DeepSeek 或中转站线路，并设置真实上游模型名、超时、提示词和扣费规则。

上游页禁止出现的内容：

- 不显示“DeepSeek 模型发布”或“中转站模型发布”。
- 不显示“客户看到的模型名”。
- 不显示“真实上游模型”。
- 不显示“线路顺序”。
- 不显示“上游附加提示词”。
- 不显示“保存线路”或“保存映射”。

新的验收标准：

- 打开 `DeepSeek 上游` 页面，只能新增、查看、修改、检查 DeepSeek 供应商。
- 打开 `中转站上游` 页面，只能新增、查看、修改、检查中转站供应商。
- 任何上游页都不能承担模型发布或模型映射动作。
- 自检脚本必须检查这个边界，一旦上游页出现模型发布或线路绑定字段，直接失败。
