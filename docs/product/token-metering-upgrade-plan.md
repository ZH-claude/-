# 统一 token 计费升级规划

更新时间：2026-06-18

## 目标

把当前“用户端按美元余额显示、商家端按人民币显示”的旧计费方式，升级为统一的“token 额度”方式。

升级后，用户不再看美元余额；用户看到的是自己还剩多少 token。商家仍然可以按人民币生成充值码，因为商家收钱和核算成本需要人民币口径。

## 核心规则

| 项目 | 新规则 |
| --- | --- |
| 用户余额 | 统一显示为 token，不再显示美元余额 |
| 充值码 | 商家按人民币生成，用户兑换后得到 token |
| 默认兑换率 | 约 8 元 = 100 万 token |
| 基准模型 | 普通 DeepSeek 按 1 倍消耗 |
| 其他模型 | 按模型倍率消耗 token |
| 失败调用 | 不扣用户 token |
| 成功调用 | 必须产生真实调用记录、真实用量记录、真实扣费流水 |
| 上游成本 | 商家端单独记录，用来算利润，不直接给用户看 |

示例：

| 模型 | 计费理解 |
| --- | --- |
| DeepSeek 普通模型 | 实际用了 1,000 token，就扣 1,000 token |
| GPT-5 | 如果倍率是 5，实际用了 1,000 token，就扣 5,000 token |
| Claude Opus 4.8 | 如果倍率是 8，实际用了 1,000 token，就扣 8,000 token |

## 两种上游要分开配置

不要把两种上游全部挤在一个页面里。商家端要拆成两个清楚入口。

操作顺序必须拆成三段：

1. `模型发布`：只准备客户看到的模型名，例如 `gpt5.5`。
2. `DeepSeek 上游` / `中转站上游`：只保存上游地址、密钥、状态和健康检查。
3. `模型映射`：才把客户模型绑定到真实上游线路，并设置真实上游模型名、线路顺序、超时、附加提示词和扣费规则。

### 入口一：DeepSeek 官方上游

用途：适合把 DeepSeek 作为基础 1 倍模型。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 名称 | 商家自己识别这条上游，例如 deepseek-v4-pro |
| 上游地址 | DeepSeek 官方接口地址 |
| 上游密钥 | 商家自己的 DeepSeek 密钥 |
| 状态 | 启用或停用 |
| 健康检查 | 检查地址和密钥是否可用 |

验收标准：

- DeepSeek 页面不发布客户模型。
- DeepSeek 页面不绑定真实上游模型名。
- DeepSeek 页面只做上游接入和检查。

### 入口二：其他中转站上游

用途：适合接入你从别的中转站拿到的价格表。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 名称 | 商家自己识别这条上游，例如中转站 1 |
| 上游地址 | 别的中转站接口地址 |
| 上游密钥 | 别的中转站给你的密钥 |
| 状态 | 启用或停用 |
| 健康检查 | 检查地址和密钥是否可用 |

中转站页面不填写客户模型名，也不填写真实模型名。输入价格、输出价格、货币、汇率、加价比例放到 `模型映射` 里，因为同一个中转站上游可能服务多个客户模型。

`模型映射` 里的倍率计算规则：

```text
1 倍 token 的成本基准 = 8 元 / 100 万 token

如果上游是人民币价格：
倍率 = 上游每 100 万 token 价格 * 1.10 / 8

如果上游是美元价格：
倍率 = 上游每 100 万 token 价格 * 汇率 * 1.10 / 8
```

图片里的示例价格：

| 项目 | 上游价格 |
| --- | --- |
| 输入 | 5 美元 / 100 万 token |
| 输出 | 30 美元 / 100 万 token |

如果商家填写汇率 7.2，那么系统应自动算出：

| 项目 | 计算 |
| --- | --- |
| 输入倍率 | 5 * 7.2 * 1.10 / 8 = 4.95 倍 |
| 输出倍率 | 30 * 7.2 * 1.10 / 8 = 29.70 倍 |

这里不写死汇率。商家输入多少汇率，系统就按多少汇率算。

## 用户端显示规则

| 页面 | 新显示 |
| --- | --- |
| 个人中心 | 剩余 token、累计消耗 token、调用次数 |
| 余额充值 | 输入充值码后显示到账多少 token |
| 令牌管理 | 显示令牌已消耗 token、剩余额度 |
| 调用日志 | 显示本次输入 token、输出 token、倍率、实际扣除 token |
| 费用说明 | 用 token 解释模型倍率，不再让用户看美元价格 |

用户端要让普通用户看得懂：

- 充值码兑换的是 token，不是美元。
- DeepSeek 是 1 倍消耗。
- 其他模型是几倍消耗，就按几倍扣。
- 用户能看到“这次为什么扣了这么多 token”。

## 商家端显示规则

| 页面 | 新显示 |
| --- | --- |
| 商家首页 | 客户剩余 token、今日消耗 token、今日预估人民币收入 |
| 充值码 | 按人民币生成充值码，同时显示可兑换 token |
| DeepSeek 上游 | 独立页面，适合 1 倍基准模型 |
| 中转站上游 | 独立页面，填写上游价格，自动加价 10% 后算倍率 |
| 模型发布 | 只管公开模型名、显示名、启停 |
| 模型映射 | 只管公开模型绑定到哪个上游、哪条线路 |
| 请求日志 | 显示真实上游、真实用量、扣除 token、上游成本、平台毛利 |

商家端必须保留人民币口径，因为充值码、成本、利润都要按人民币核算。

## 最小任务拆分

### B01 规则锁定与页面说明

状态：待开始

范围：

- 把旧的美元余额说明改为 token 余额说明。
- 明确商家端人民币、用户端 token 的分工。
- 明确 8 元约 100 万 token 的默认兑换率。

完成指标：

- 用户端不再出现“美元余额”作为目标计费口径。
- 商家端文案能解释“人民币生成充值码，用户得到 token”。
- 文档、页面说明和后续任务口径一致。

### B02 充值码改为发放 token

状态：待开始

范围：

- 商家输入人民币金额。
- 系统按兑换率算出 token。
- 用户兑换后增加 token。
- 历史流水仍能追踪这张充值码来自多少人民币。

完成指标：

- 8 元充值码兑换后，用户得到约 100 万 token。
- 充值码状态、用户余额、流水三处一致。
- 同一张充值码不能重复兑换。

### B03 模型倍率改为 token 消耗

状态：待开始

范围：

- DeepSeek 默认 1 倍。
- 其他模型可设置输入倍率和输出倍率。
- 调用完成后按真实输入、真实输出、倍率扣 token。

完成指标：

- DeepSeek 1,000 token 调用扣 1,000 token。
- GPT-5 设置 5 倍时，1,000 token 调用扣 5,000 token。
- Claude Opus 4.8 设置 8 倍时，1,000 token 调用扣 8,000 token。
- 调用失败不扣。

### B04 中转站上游价格自动换算倍率

状态：待开始

范围：

- 商家输入上游价格。
- 商家输入汇率。
- 系统默认加价 10%。
- 系统自动算出输入倍率和输出倍率。

完成指标：

- 输入价格 5 美元 / 100 万 token、汇率 7.2 时，输入倍率显示 4.95。
- 输出价格 30 美元 / 100 万 token、汇率 7.2 时，输出倍率显示 29.70。
- 改汇率或加价比例后，倍率实时变化。

### B05 双上游独立页面

状态：进行中

范围：

- DeepSeek 官方上游单独一个入口。
- 其他中转站上游单独一个入口。
- 上游页面只能新增、查看、修改、检查上游。
- 上游页面不能发布客户模型，也不能绑定真实上游模型。
- 模型发布页只发布客户模型。
- 模型映射页才绑定客户模型和上游线路。

完成指标：

- 商家能清楚知道自己正在配置哪一种上游。
- 两种上游各自能新增、查看、修改、检查。
- 页面不再让商家在一个长页面里来回找。
- 不会把 DeepSeek 上游、其它中转站上游、客户模型发布、线路绑定混在一起。

### B06 消费日志和对账

状态：待开始

范围：

- 记录真实输入 token、输出 token。
- 记录模型倍率。
- 记录实际扣除 token。
- 记录上游成本和平台毛利。

完成指标：

- 用户能看到每次扣了多少 token。
- 商家能看到本次请求成本和毛利。
- 令牌消费、用户余额、请求日志、流水四处能互相对上。

### B07 旧数据迁移和兼容

状态：待开始

范围：

- 当前已有的美元余额不能直接丢。
- 需要按明确规则转成 token。
- 转换前后要有审计记录。

完成指标：

- 现有用户余额能转换为 token。
- 转换后不会出现负数。
- 历史记录仍能解释当时是旧规则还是新规则。

## 必须先确认的设置

下面这些不要写死，必须做成商家可改：

| 设置 | 默认建议 |
| --- | --- |
| 人民币兑换 token | 8 元 = 100 万 token |
| 中转站价格加价比例 | 10% |
| 美元兑人民币汇率 | 商家手动填写 |
| DeepSeek 倍率 | 1 |
| 其他模型倍率 | 商家输入价格后自动计算，也允许商家手动覆盖 |

## 不能做的事

- 不能再让用户端继续围绕美元余额理解消费。
- 不能把 DeepSeek 和其他中转站上游挤在一个页面里。
- 不能在没有真实用量的情况下假装扣费成功。
- 不能把上游密钥、用户令牌、数据库密码写进文档。
- 不能只改页面显示，后台账本仍按旧美元逻辑扣费。

## 2026-06-19 流程纠偏

正确业务顺序已经统一为三步：

1. 先发布客户模型：例如先在 `模型发布` 页面发布客户看到的 `gpt5.5`，这里只决定用户端能看到什么模型名和状态。
2. 再接入上游供应商：`DeepSeek 上游` 只保存 DeepSeek 的上游地址、密钥、状态和健康检查；`中转站上游` 只保存其他中转站的上游地址、密钥、状态和健康检查。
3. 最后绑定上游线路：在 `模型映射` 页面把 `gpt5.5` 绑定到一个或多个上游线路，例如 DeepSeek 线路或中转站 1 线路。这里填写真实上游模型名、线路顺序、超时时间、附加提示词、线路计费方式和是否支持流式输出。

边界规则：

- 上游页面不能直接发布客户模型，也不能替代模型发布页面。
- 一个客户模型可以有多条上游线路，系统按线路顺序尝试；前一条超时或失败时，才切换到下一条。
- `gpt5.5` 这类客户模型是对用户展示的名称；真实请求走哪个上游、真实上游模型叫什么，由绑定线路决定。
- DeepSeek 上游和其他中转站上游可以有不同计费方式，但最终都必须折算到统一 token 余额。
- 后续继续开发时，任何页面、接口、自检脚本都必须按以上顺序验收，不能再把“上游接入”和“客户模型发布”混在同一个入口里。

## 2026-06-19 二次纠偏：先模型，后上游，最后线路

这次纠偏后的目标不是在上游页面直接发布模型，而是把三个东西彻底分开：

| 步骤 | 商家看到的动作 | 保存的真实含义 | 验收标准 |
| --- | --- | --- | --- |
| 1 | 发布客户模型，例如 `gpt5.5` | 只定义用户端可见的模型商品 | 用户端只在绑定可用线路后看到该模型 |
| 2 | 接入 DeepSeek 上游或中转站上游 | 只保存供应商地址、密钥、状态、健康检查 | 上游列表能查看、修改、检查，不发布客户模型 |
| 3 | 模型映射选择上游线路 | 把客户模型绑定到具体上游、真实上游模型、线路顺序、超时和提示词 | 一个客户模型最多三条线路，按顺序失败切换 |
| 4 | 在线路上配置扣费 | DeepSeek 线路按倍率扣 token；中转站线路按上游价格加 10% 换算 token | 同一个 `gpt5.5` 走不同上游时，也能按对应线路正确扣 token |

后续最小任务：

- [x] 任务 A：把页面入口重新收口，`模型发布` 只管客户模型名、展示名、状态；上游接入页只管供应商；`模型映射` 才管选择上游。
- [x] 任务 B：把线路计费落到每条上游线路上，避免一个 `gpt5.5` 同时有 DeepSeek 和中转站线路时计费混乱。
- [ ] 任务 C：DeepSeek 线路支持更清楚的倍率入口，例如 `gpt5.5` 填 5 倍，实际 1,000 token 按 5,000 token 扣。
- [ ] 任务 D：中转站线路继续打磨输入价、输出价、币种、汇率、加价百分比的页面说明，让系统自动换算出的统一 token 扣费更容易检查。
- [ ] 任务 E：真实调用后继续补强用户余额、令牌消耗、请求日志、商家审计记录的完整对账页面。
