# 统一 token 计费升级规划

更新时间：2026-06-18

## 目标

把当前“用户端按美元余额显示、商家端按人民币显示”的旧计费方式，升级为统一的“基础 token 额度”方式。

升级后，用户不再看美元余额；用户看到的是自己还剩多少“基础 token”。商家仍然可以按人民币生成充值码，因为商家收钱和核算成本需要人民币口径。

## 核心规则

| 项目 | 新规则 |
| --- | --- |
| 用户余额 | 统一显示为基础 token，不再显示美元余额 |
| 充值码 | 商家按人民币生成，用户兑换后得到基础 token |
| 默认兑换率 | 约 8 元 = 100 万基础 token |
| 基准模型 | 普通 DeepSeek 按 1 倍消耗 |
| 其他模型 | 按模型倍率消耗基础 token |
| 失败调用 | 不扣用户 token |
| 成功调用 | 必须产生真实调用记录、真实用量记录、真实扣费流水 |
| 上游成本 | 商家端单独记录，用来算利润，不直接给用户看 |

示例：

| 模型 | 计费理解 |
| --- | --- |
| DeepSeek 普通模型 | 实际用了 1,000 token，就扣 1,000 基础 token |
| GPT-5 | 如果倍率是 5，实际用了 1,000 token，就扣 5,000 基础 token |
| Claude Opus 4.8 | 如果倍率是 8，实际用了 1,000 token，就扣 8,000 基础 token |

## 两种上游要分开配置

不要把两种上游全部挤在一个页面里。商家端要拆成两个清楚入口。

### 入口一：DeepSeek 官方上游

用途：适合把 DeepSeek 作为基础 1 倍模型。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 上游地址 | DeepSeek 官方接口地址 |
| 上游密钥 | 商家自己的 DeepSeek 密钥 |
| 公开模型名 | 给用户看到的模型名 |
| 真实模型名 | 实际请求上游时使用的模型名 |
| 模型倍率 | 默认 1，可手动改 |
| 超时时间 | 超过多少秒换下一条线路 |
| 附加提示词 | 每次请求上游前追加的商家提示词 |

验收标准：

- DeepSeek 默认可以按 1 倍扣基础 token。
- 失败、超时、上游报错时不扣用户 token。
- 成功调用后，用户余额、令牌消费、调用日志三处数字一致。

### 入口二：其他中转站上游

用途：适合接入你从别的中转站拿到的价格表。

商家需要填写：

| 字段 | 说明 |
| --- | --- |
| 上游地址 | 别的中转站接口地址 |
| 上游密钥 | 别的中转站给你的密钥 |
| 公开模型名 | 给用户看到的模型名 |
| 真实模型名 | 实际请求上游时使用的模型名 |
| 输入价格 | 上游给你的输入价格 |
| 输出价格 | 上游给你的输出价格 |
| 货币 | 上游价格是美元还是人民币 |
| 汇率 | 如果上游是美元价格，需要商家填写汇率 |
| 加价比例 | 默认 10% |
| 系统算出的倍率 | 系统按上面信息自动算，不让用户自己猜 |
| 超时时间 | 超过多少秒换下一条线路 |
| 附加提示词 | 每次请求上游前追加的商家提示词 |

倍率计算规则：

```text
1 倍基础 token 的成本基准 = 8 元 / 100 万 token

如果上游是人民币价格：
倍率 = 上游每 100 万 token 价格 * 1.10 / 8

如果上游是美元价格：
倍率 = 上游每 100 万 token 价格 * 汇率 * 1.10 / 8
```

图片里的示例价格：

| 项目 | 上游价格 |
| --- | --- |
| 输入 | 5 美元 / 100 万 token |
| 输出 | 30 美元 / 100 万 token |

如果商家填写汇率 7.2，那么系统应自动算出：

| 项目 | 计算 |
| --- | --- |
| 输入倍率 | 5 * 7.2 * 1.10 / 8 = 4.95 倍 |
| 输出倍率 | 30 * 7.2 * 1.10 / 8 = 29.70 倍 |

这里不写死汇率。商家输入多少汇率，系统就按多少汇率算。

## 用户端显示规则

| 页面 | 新显示 |
| --- | --- |
| 个人中心 | 剩余基础 token、累计消耗基础 token、调用次数 |
| 余额充值 | 输入充值码后显示到账多少基础 token |
| 令牌管理 | 显示令牌已消耗基础 token、剩余额度 |
| 调用日志 | 显示本次输入 token、输出 token、倍率、实际扣除基础 token |
| 费用说明 | 用基础 token 解释模型倍率，不再让用户看美元价格 |

用户端要让普通用户看得懂：

- 充值码兑换的是 token，不是美元。
- DeepSeek 是 1 倍消耗。
- 其他模型是几倍消耗，就按几倍扣。
- 用户能看到“这次为什么扣了这么多 token”。

## 商家端显示规则

| 页面 | 新显示 |
| --- | --- |
| 商家首页 | 客户剩余基础 token、今日消耗基础 token、今日预估人民币收入 |
| 充值码 | 按人民币生成充值码，同时显示可兑换基础 token |
| DeepSeek 上游 | 独立页面，适合 1 倍基准模型 |
| 中转站上游 | 独立页面，填写上游价格，自动加价 10% 后算倍率 |
| 模型发布 | 只管公开模型名、显示名、倍率、启停 |
| 模型映射 | 只管公开模型绑定到哪个上游、哪条线路 |
| 请求日志 | 显示真实上游、真实用量、扣除 token、上游成本、平台毛利 |

商家端必须保留人民币口径，因为充值码、成本、利润都要按人民币核算。

## 最小任务拆分

### B01 规则锁定与页面说明

状态：待开始

范围：

- 把旧的美元余额说明改为基础 token 余额说明。
- 明确商家端人民币、用户端 token 的分工。
- 明确 8 元约 100 万基础 token 的默认兑换率。

完成指标：

- 用户端不再出现“美元余额”作为目标计费口径。
- 商家端文案能解释“人民币生成充值码，用户得到 token”。
- 文档、页面说明和后续任务口径一致。

### B02 充值码改为发放基础 token

状态：待开始

范围：

- 商家输入人民币金额。
- 系统按兑换率算出基础 token。
- 用户兑换后增加基础 token。
- 历史流水仍能追踪这张充值码来自多少人民币。

完成指标：

- 8 元充值码兑换后，用户得到约 100 万基础 token。
- 充值码状态、用户余额、流水三处一致。
- 同一张充值码不能重复兑换。

### B03 模型倍率改为基础 token 消耗

状态：待开始

范围：

- DeepSeek 默认 1 倍。
- 其他模型可设置输入倍率和输出倍率。
- 调用完成后按真实输入、真实输出、倍率扣基础 token。

完成指标：

- DeepSeek 1,000 token 调用扣 1,000 基础 token。
- GPT-5 设置 5 倍时，1,000 token 调用扣 5,000 基础 token。
- Claude Opus 4.8 设置 8 倍时，1,000 token 调用扣 8,000 基础 token。
- 调用失败不扣。

### B04 中转站上游价格自动换算倍率

状态：待开始

范围：

- 商家输入上游价格。
- 商家输入汇率。
- 系统默认加价 10%。
- 系统自动算出输入倍率和输出倍率。

完成指标：

- 输入价格 5 美元 / 100 万 token、汇率 7.2 时，输入倍率显示 4.95。
- 输出价格 30 美元 / 100 万 token、汇率 7.2 时，输出倍率显示 29.70。
- 改汇率或加价比例后，倍率实时变化。

### B05 双上游独立页面

状态：待开始

范围：

- DeepSeek 官方上游单独一个入口。
- 其他中转站上游单独一个入口。
- 模型发布和模型映射不再挤在同一个大页面里。

完成指标：

- 商家能清楚知道自己正在配置哪一种上游。
- 两种上游各自能新增、查看、修改、检查。
- 页面不再让商家在一个长页面里来回找。

### B06 消费日志和对账

状态：待开始

范围：

- 记录真实输入 token、输出 token。
- 记录模型倍率。
- 记录实际扣除基础 token。
- 记录上游成本和平台毛利。

完成指标：

- 用户能看到每次扣了多少基础 token。
- 商家能看到本次请求成本和毛利。
- 令牌消费、用户余额、请求日志、流水四处能互相对上。

### B07 旧数据迁移和兼容

状态：待开始

范围：

- 当前已有的美元余额不能直接丢。
- 需要按明确规则转成基础 token。
- 转换前后要有审计记录。

完成指标：

- 现有用户余额能转换为基础 token。
- 转换后不会出现负数。
- 历史记录仍能解释当时是旧规则还是新规则。

## 必须先确认的设置

下面这些不要写死，必须做成商家可改：

| 设置 | 默认建议 |
| --- | --- |
| 人民币兑换基础 token | 8 元 = 100 万基础 token |
| 中转站价格加价比例 | 10% |
| 美元兑人民币汇率 | 商家手动填写 |
| DeepSeek 倍率 | 1 |
| 其他模型倍率 | 商家输入价格后自动计算，也允许商家手动覆盖 |

## 不能做的事

- 不能再让用户端继续围绕美元余额理解消费。
- 不能把 DeepSeek 和其他中转站上游挤在一个页面里。
- 不能在没有真实用量的情况下假装扣费成功。
- 不能把上游密钥、用户令牌、数据库密码写进文档。
- 不能只改页面显示，后台账本仍按旧美元逻辑扣费。
