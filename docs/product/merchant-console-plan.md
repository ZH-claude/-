# 商家端独立化最小任务计划

## 目标

把现在“后台账号登录后仍像用户端”的问题修掉。MVP 不拆成两个部署，也不做第二套登录系统；采用同一个 Web、同一个登录入口，根据账号角色进入不同工作台：

- 普通用户 `role=user`：进入用户端，管理账户、余额、令牌、日志、价格、通知。
- 商家/后台账号 `role=admin`：进入商家端，管理用户、充值码、模型、上游、公告、审计、服务状态。

当前数据库只有 `USER` 和 `ADMIN` 两种角色。MVP 阶段“商家端账号”使用 `ADMIN` 角色承载；如果后续要做多商户 SaaS，再新增 `MERCHANT` 角色、商户表和租户隔离。

## 总体验收标准

- 商家账号登录后默认进入 `/merchant` 或后台工作台，不再进入用户中心。
- 用户端和商家端的顶部栏、侧边栏、首页指标和主要菜单明显不同。
- 普通用户不能看到商家端入口，直接访问商家端页面时被拒绝或跳走。
- 商家端所有数字和列表来自真实 API/数据库；没有 mock、假数据、硬编码业务结果。
- 每完成一个最小任务，都执行一次 QA + Review，自检通过后再进入下一个任务。

## 每个任务完成后的固定自检

1. 工程检查：
   - `npm run typecheck`
   - 涉及构建或路由时运行 `npm run build`
2. 真实数据检查：
   - 使用普通用户账号验证用户端仍可用。
   - 使用 `merchant_test_1 / merchant200611` 验证商家端真实登录。
   - 对新增/改动页面检查对应后端接口返回真实数据库数据。
3. QA：
   - 按用户视角在浏览器跑一遍核心路径。
   - 检查页面截图、控制台错误、跳转、空状态、权限错误态。
4. Review：
   - 做代码审查，重点看权限隔离、数据泄漏、假数据、回归风险、兼容性。
5. 记录：
   - 在对应任务后打勾。
   - 写明验证命令、账号、页面、发现的问题和修复结果。
   - 提交并推送 GitHub。

## 商用升级计划：多商家上游密钥隔离

当前 MVP 是单平台老板模式：你在商家端配置上游密钥，所有客户走平台统一上游。商用版如果要开放多个商家入驻，必须新增多商家隔离，不应直接在 M06 中硬塞。

### C01 商家归属与客户归属

状态：待开始

范围：
- 为商家账号建立独立归属关系。
- 明确客户属于哪个商家，客户密钥也跟随该商家。
- 保留平台老板账号的全局管理能力。

验收指标：
- 商家 A 看不到商家 B 的客户、余额、上游配置和日志。
- 平台老板可以统一查看和管理。

### C02 商家独立上游密钥

状态：待开始

范围：
- 每个商家可保存自己的上游地址和上游密钥。
- 上游密钥仍必须加密保存、只显示脱敏预览。
- 同一公开模型可以映射到不同商家的不同上游模型。

验收指标：
- 商家 A/B 分别配置不同上游密钥后，互相不可见。
- 禁用某个商家的上游，只影响该商家的客户。

### C03 客户请求按商家转发

状态：待开始

范围：
- 客户发起请求时，系统按客户归属找到对应商家，再选择该商家的上游密钥转发。
- 调用日志记录商家、客户、模型、上游和密钥指纹，不记录明文密钥。

验收指标：
- 客户 A 的请求只能用商家 A 的上游密钥。
- 客户 B 的请求只能用商家 B 的上游密钥。
- 上游返回失败时，错误归属到对应商家，不影响其他商家。

### C04 商用闭环验收

状态：待开始

范围：
