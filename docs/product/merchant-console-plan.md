# 商家端独立化最小任务计划

## 目标

把现在“后台账号登录后仍像用户端”的问题修掉。MVP 不拆成两个部署，也不做第二套登录系统；采用同一个 Web、同一个登录入口，根据账号角色进入不同工作台：

- 普通用户 `role=user`：进入用户端，管理账户、余额、令牌、日志、价格、通知。
- 商家/后台账号 `role=admin`：进入商家端，管理用户、充值码、模型、上游、公告、审计、服务状态。

当前数据库只有 `USER` 和 `ADMIN` 两种角色。MVP 阶段“商家端账号”使用 `ADMIN` 角色承载；如果后续要做多商户 SaaS，再新增 `MERCHANT` 角色、商户表和租户隔离。

## 总体验收标准

- 商家账号登录后默认进入 `/merchant` 或后台工作台，不再进入用户中心。
- 用户端和商家端的顶部栏、侧边栏、首页指标和主要菜单明显不同。
- 普通用户不能看到商家端入口，直接访问商家端页面时被拒绝或跳走。
- 商家端所有数字和列表来自真实 API/数据库；没有 mock、假数据、硬编码业务结果。
- 每完成一个最小任务，都执行一次 QA + Review，自检通过后再进入下一个任务。

## 每个任务完成后的固定自检

1. 工程检查：
   - `npm run typecheck`
   - 涉及构建或路由时运行 `npm run build`
2. 真实数据检查：
   - 使用普通用户账号验证用户端仍可用。
   - 使用 `merchant_test_1 / merchant200611` 验证商家端真实登录。
   - 对新增/改动页面检查对应后端接口返回真实数据库数据。
3. QA：
   - 按用户视角在浏览器跑一遍核心路径。
   - 检查页面截图、控制台错误、跳转、空状态、权限错误态。
4. Review：
   - 做代码审查，重点看权限隔离、数据泄漏、假数据、回归风险、兼容性。
5. 记录：
   - 在对应任务后打勾。
   - 写明验证命令、账号、页面、发现的问题和修复结果。
   - 提交并推送 GitHub。

## 最小任务拆分

### M01 登录后角色分流

状态：待开始

范围：
- 修改登录成功后的跳转逻辑。
- `role=user` 跳 `/account/profile`。
- `role=admin` 跳商家端入口，建议先用 `/merchant`，并保留 `/admin` 兼容跳转。

验收指标：
- `merchant_test_1` 登录后自动进入商家端。
- 普通测试用户登录后仍进入用户中心。
- 刷新页面后登录态保持正常。
- 不改动后端权限模型，不引入假角色。

QA/Review 重点：
- 登录成功但角色缺失时的 fallback。
- 401/403 是否正确回登录页或错误页。
- 用户端现有登录流程不回归。

### M02 商家端 Shell 与固定导航

状态：待开始

范围：
- 新增 `MerchantShell` 或等效组件。
- 商家端使用独立顶部栏和左侧固定栏。
- 固定栏包含：首页、用户、充值码、分组状态、令牌/日志入口、绘图日志、上游/模型、公告、审计、服务状态。
- 删除商家端里的普通用户中心侧栏。

验收指标：
- 商家端视觉结构与用户端明显不同。
- 左侧栏和顶部栏固定，不出现页面像普通用户端一样单列卡片堆叠。
- 移动端不重叠，菜单可横向/抽屉式访问。
- 退出、刷新、当前账号显示可用。

QA/Review 重点：
- 布局在 1920、1366、移动端不溢出。
- 导航 active 状态准确。
- 没有把用户端菜单暴露给商家端。

### M03 商家端入口 Dashboard

状态：待开始

范围：
- 新增 `/merchant` 首页。
- 后端复用或新增真实汇总接口，展示用户数、余额总额、今日调用、今日消费、上游健康、模型数量、充值码状态、最近告警。
- 无数据时展示真实空状态，不放模拟数据。

验收指标：
- Dashboard 所有数字可追溯到真实数据库查询。
- 管理员账号可访问，普通用户访问被拒绝。
- 没有上游、没有模型、没有日志时显示空状态而不是 500。

QA/Review 重点：
- 聚合查询是否过滤软删除用户。
- 是否暴露上游 API Key、token hash、充值码 hash。
- 空数据库和已有数据两种场景都能加载。

### M04 商家端用户管理页

状态：待开始

范围：
- 从当前 `/admin` 大页面拆出用户管理页。
- 支持用户列表、角色/状态/分组/余额/最后登录查看。
- 支持给用户分组。
- 普通用户不能访问。

验收指标：
- 列表来自 `/admin/users` 真实数据。
- 分组变更后数据库更新，刷新后仍存在。
- 用户密码哈希、会话 token、API token hash 不出现在前端响应。

QA/Review 重点：
- 越权访问。
- 分组变更失败时的错误提示。
- 大量用户时分页或限制清晰。

### M05 商家端充值码管理页

状态：待开始

范围：
- 从当前 `/admin` 大页面拆出充值码页面。
- 支持生成充值码、一次性复制本次生成结果、查看充值码状态、禁用未使用充值码。

验收指标：
- 生成的充值码只在创建响应中出现一次。
- 列表不显示明文充值码和 hash。
- 用户兑换后状态变为 used，余额真实增加。

QA/Review 重点：
- 防止明文充值码落入日志或审计响应。
- 禁用已使用充值码应失败或无副作用。
- 余额事务一致性。

### M06 商家端模型与上游配置页

状态：待开始

范围：
- 拆出模型价格、分组授权、上游 Provider、上游模型映射、健康检查。
- 保留现有 SSRF 防护和上游 Key 加密逻辑。

验收指标：
- 新增模型后用户端价格页/可用模型页能看到真实变化。
- 新增上游映射后 `/v1/models` 返回对应真实模型。
- 健康检查不允许本地/私网/metadata 地址。

QA/Review 重点：
- 上游 Key 不泄漏。
- 模型名、URL、倍率校验。
- 上游异常不拖垮页面。

### M07 商家端公告与审计页

状态：待开始

范围：
- 拆出公告发布、公告记录、后台审计、安全审计。
- 审计页只读，支持分页。

验收指标：
- 发布公告后用户首页真实可见。
- 草稿和归档不出现在用户公开公告里。
- 审计日志响应脱敏。

QA/Review 重点：
- 普通用户不能访问审计接口。
- 审计数据不暴露 `password`、`tokenHash`、`encryptedApiKey`、`codeHash`、连接串。

### M08 商家端服务状态与日志入口

状态：待开始

范围：
- 商家端增加服务状态、请求日志、绘图日志入口。
- 删除没有实际上游支持的“异步任务”独立菜单；保留绘图日志只显示真实 `kind=image` 任务。

验收指标：
- 服务状态页展示 API/Web/上游真实状态。
- 日志页仍按当前账号权限正确读取。
- 异步任务入口不再误导用户。

QA/Review 重点：
- 500 错误定位并修掉，不能用隐藏错误代替。
- 没有数据时显示空状态。
- 菜单与实际可用能力一致。

### M09 角色隔离与端到端回归

状态：待开始

范围：
- 为普通用户和商家账号各跑一条完整流程。
- 补充必要的脚本级 smoke test。
- 检查所有商家端 API 都由 `AuthGuard + AdminGuard` 保护。

验收指标：
- 普通用户无法访问任一商家端页面/API。
- 商家账号可访问全部商家端页面。
- 用户端核心路径不回归：账户、充值、令牌、日志、价格、模型、通知。
- 商家端核心路径不回归：用户、充值码、模型、上游、公告、审计、服务状态。

QA/Review 重点：
- 浏览器真实流程截图。
- 控制台无前端错误。
- 后端无 500。
- `npm run typecheck` 和 `npm run build` 通过。

### M10 收尾、文档与部署检查

状态：待开始

范围：
- 更新 README、部署文档、测试账号说明。
- 标注 MVP 商家端和未来真正 `MERCHANT` 多商户能力的边界。
- 推送 GitHub。

验收指标：
- 文档清楚说明同一个 Web 按角色分流。
- 测试账号不硬编码进生产脚本，密码只通过环境变量或用户本地说明提供。
- 云服务器部署步骤不需要额外第二个前端。

QA/Review 重点：
- 文档与代码一致。
- 不提交 `.env`、真实密钥、真实上游 Key。
- 部署 smoke test 仍可运行。

## 推荐执行顺序

先做 M01-M03。完成后你马上能看到“商家账号进的是另一个工作台”，不会再觉得一模一样。然后再拆 M04-M08，把当前 `/admin` 大页面逐步拆成真正商家端模块。最后做 M09-M10 做全量回归和交付。
